# 22 — 自製譜面系統

> 來源:使用者需求(2026-07-12,backlog 記錄)。
> **狀態:backlog,大題、方向全待 grill。是新增能力,會擴張架構邊界,務必先 grill 定形態再動手。**

## 問題 / 動機

目前所有譜面都來自 Beat Saber(內建 / zip / BSR),經 `compileChart` 轉成 TypingChart。缺一條「自己做譜」的路:讓使用者不依賴既有 Beat Saber 圖,也能產生要練的內容(自訂節奏 / 針對性練特定鍵或字 / 匯入一段文字當練習)。這開啟「Beat Typer 作為獨立打字練習工具」的可能,而不只是 Beat Saber 譜面的轉換器。

## 核心待定(grill 時逐一決)

這題最大的不確定在**產物接到架構的哪一層**,先定這個再談 UX:

- **產物層級**(關鍵):
  - **A. 直接產 TypingChart**:自製系統輸出已攤平的 `{tSec, key, kind, hand/finger/bank}` 時間軸,繞過 `compileChart`。最直接,但會出現「不經唯一正規化點」的第二條產出路徑,需想清楚可決定性 / replay / 成績身分怎麼算。
  - **B. 產 Beat-Saber-like RawMapFiles**:自製系統輸出原始譜面檔(如同一個 `ChartSource`),仍走 `compileChart` 正規化。與現有架構最一致(自製只是「第四種來源」),但要遷就 Beat Saber 資料模型的表達力。
- **輸入形態**:視覺編輯器(放音符)/ 文字轉譜(打一段字 → 自動排成節奏)/ 節奏模式產生器(選 BPM + 鍵群 + 模式)?每種的複雜度與受眾差很多。
- **音訊**:自製譜配什麼聲音?無音樂(純節拍器 / tick)/ 上傳音檔 / 用內建合成節拍?判定主時鐘目前綁 `player.positionSec`,無音樂時要有替代時鐘。
- **儲存 / 分享**:自製譜存哪(localStorage / 匯出檔)?要不要能分享(對應 issue 16 的 BSR 那種一鍵載入)?
- **成績身分**:自製譜怎麼算 `songKey`(issue 18 用難度檔內容雜湊)——自製產物要有等價的內容雜湊來記成績。
- **鍵群 / 映射**:自製時使用者是直接指定鍵,還是仍走鍵指派 / 鍵群(issue 11/15)?

## What to build(待 grill 後才定;先不寫死)

依上面「產物層級」的抉擇分岔,可能是:一個新的 `ChartSource`(B 路)、或一個直達 TypingChart 的產生器 + 對應的成績身分 / 時鐘處理(A 路),外加一套建立 / 編輯 / 試玩的 UI。

## Acceptance criteria(暫,全待 grill 收斂)

- [ ] 能不依賴 Beat Saber 圖,建立並遊玩一份自製練習譜
- [ ] 產物接入既有遊玩 / 判定 / 成績路徑(身分、可決定性、時鐘都有交代)
- [ ] compileChart / judge 的純函式與測試接縫紀律不被破壞
- [ ] 自製譜可存 / 重開(至少本機)

## 關聯 / Blocked by

- 觸及 issue 06(遊玩)、07/08(判定)、11/15(映射 / 鍵群)、12(持久化)、18(成績身分)、16(若做分享)。
- 與 issue 05(CustomLevels 資料夾來源)不同:05 仍是讀 Beat Saber 圖,本議是「無中生有做譜」。

## Out of scope(暫定,grill 再議)

- 完整的 Beat Saber 關卡編輯器(切法方向 / 牆 / 燈光等本作不用的維度)。
- 譜面市集 / 伺服器端(先本機 / 檔案分享即可)。
