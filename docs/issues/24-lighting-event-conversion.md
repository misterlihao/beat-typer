# 24 — Beat Saber 譜面燈光效果轉換

> 來源:使用者需求(2026-07-12,backlog 記錄)。
> **狀態:backlog,大題、方向待 grill。牽動解析 + 渲染;是否經 compileChart 正規化為先決題。**

## 問題 / 動機

Beat Saber 譜面除了音符,還帶**燈光 / 事件**資料(v2 `_events`、v3 `basicBeatmapEvents` 及新版燈光),原作靠它做全場燈光秀、與音樂同步。本作目前完全忽略這塊,場景是靜態的。把譜面燈光轉成本作的視覺律動(背景 / 環境燈隨歌閃動、換色、淡入淡出),能讓畫面跟著音樂呼吸,大幅提升沉浸感,且是「免費」的內容——資料已在譜裡。

## 核心待定(grill 時逐一決)

- **接哪一層(關鍵)**:
  - **A. 經 compileChart 正規化**:產出一條與 TypingChart 並行的「燈光時間線」(標準化的 `{tSec, 群組, 動作, 顏色}`),維持「唯一正規化點」紀律、可測、時間對齊沿用 BPM 時間線(issue 10)。較重但架構乾淨。
  - **B. 渲染層薄解析**:highway 直接讀事件、即時演算,不進 compile。快,但把譜面格式細節與時間換算漏進渲染薄層。
- **對應到什麼視覺**:Beat Saber 有多個燈光群組(後方 / 側邊 / 環等)與動作(on / off / flash / fade)、紅藍雙色。要對到本作哪些元素(背景、走廊氛圍光、場景 issue 20)?**不可打到音符 / 判定格的紅藍與判定四色**,避免混淆。
- **涵蓋範圍**:v2 `_events` 與 v3 事件格式差異大、新版燈光(v3 lighting)極複雜。先支援到哪個程度(如只取基本 on/flash + 顏色,忽略進階群組變換)?
- **強度 / 開關**:燈光可能很晃眼或干擾判讀,需可調 / 可關,並以 **issue 12** 持久化。

## What to build(待 grill 後才定)

依「接哪一層」分岔:A 路要新增燈光解析 + 正規化(純函式、可測)+ 渲染消費;B 路則是 highway 內的薄解析 + 即時演出。兩路都需一組「燈光 → 場景視覺」的對應表,並嚴守與判定色的區隔。

## Acceptance criteria(暫,全待 grill 收斂)

- [ ] 背景 / 環境視覺隨譜面燈光事件與音樂同步律動
- [ ] 燈光視覺不干擾音符 / 判定辨識(顏色 / 位置 / 亮度有區隔)
- [ ] 時間對齊沿用 BPM 時間線(issue 10),與音符同一時鐘
- [ ] 可調強度 / 可關(issue 12 持久化);無燈光資料的譜優雅退化(靜態或預設律動)
- [ ] (若走 A)燈光正規化為純函式,有 vitest fixtures;compileChart/judge 既有紀律不破

## 關聯 / Blocked by

- 依賴 issue 06(渲染)、issue 10(BPM 時間線 / beat→秒換算)。
- 與 issue 20(場景)、issue 23(空段落演出)天然聯動:燈光正是場景與空段演出的驅動來源。
- 與音符映射(compile)正交:燈光不影響 TypingChart、判定或計分。

## Out of scope

- 還原 Beat Saber 完整燈光語意(所有群組 / 進階變換 / rotation events)。
- 燈光編輯(見 issue 22 自製譜面系統,另議)。
