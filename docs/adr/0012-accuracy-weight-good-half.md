# 準確率權重:Perfect 1 / Good 0.5 / Miss 0

**Status:** accepted

準確率(結算與評級的唯一依據)= Σ權重 ÷ 音符數,權重為 **Perfect 1、Good 0.5、Miss 0**。實作在 [judge.ts](../../src/judge/judge.ts) 的 `WEIGHT` 常數與 `summary()`;評級門檻(S≥.95 / A≥.85 / B≥.7 / C≥.5 / D)吃這個準確率。分母是「已解算音符數」(歌播畢=全部音符),多餘按鍵(extras)不進分子也不進分母。

## 為何記錄

Good 只給半分是刻意選擇,且對評級結果影響很大——一個未來讀者看到「全 Good 只有 C」會想問為什麼。這是真實取捨:

1. **半分(採用)**:節奏抓得鬆(落在 Good 窗而非 Perfect 窗)只給一半分。對「打字/節奏練習」定位,把「準時」和「大致對」明確分級,逼玩家往 Perfect 收斂。代價:對休閒玩家偏嚴,全 Good 一場只有 C。
2. **滿分(否決)**:Good=1 會讓準確率退化成「命中率」,失去區分節奏精度的能力,S 太好拿,評級失去練習導向的訊號。

## 關鍵推理

- 權重只在 `judge`(純函式接縫)一處定義,結算層只讀 `summary`、不重算——CONTEXT「結算只呈現不計算」的紀律靠這條落實。
- 門檻數字(.95/.85/…)是可調旋鈕、易逆轉,故**不**進 ADR;真正的取捨是「Good 半分」這個權重語義,只記這條。
- extras 不計入分母:多餘按鍵「完全不罰」(見 CONTEXT 多餘按鍵),否則亂敲會稀釋準確率、與「不罰」矛盾。

## 被否決 / 排除

- **Good=1(命中率式準確率)**:見上。
- **加入 extras 懲罰準確率**:違反「多餘按鍵不罰」的既定契約。
- **為門檻數字寫 ADR**:門檻是旋鈕不是架構承諾,改一個函式即可,不符 ADR 門檻。

## 連帶影響

- 結算畫面(issue 09)顯示的準確率%、評級皆直接來自此權重;調權重會同時改動即時 HUD 與結算,兩處一致(單一真相)。
