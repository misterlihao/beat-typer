# 欄→手指採空間順序保留,並以「鍵盤形高速公路」渲染

**Status:** accepted

音符的欄(col0~3,玩家視角由左到右)映射到該手手指時,採用**空間順序保留**:同一手內,欄由左到右 = 該手手指由左到右(故左手 col3→食指、右手 col0→食指,兩食指皆在內側)。3D 高速公路則以**每根手指一條道、道的左右順序照實體鍵盤排**來呈現。

## 為何記錄

初看會以為該用「人體工學」原則,把 Beat Saber 最自然/最頻繁的欄(左 col1、右 col2)綁到最強的食指定位鍵 F/J。我們**沒有**這樣做,理由不直觀,值得留痕。

## 關鍵推理

- 玩家在遊戲中只看到**飛來的字母**,永遠感知不到音符在原譜的欄位。因此「映射是否符合直覺/好預測」對玩家無影響——那只是實作端的一次性成本。
- 所以欄→手指的排列,唯一能被玩家感知的前提是**空間化渲染**:當每根手指一條道、且道照鍵盤排列時,空間順序映射會讓「音符在畫面上的左右位置 = 該鍵在鍵盤上的左右位置」,高速公路變成一張「會動的鍵盤」,玩家可據道預先移動手指。
- 「自然欄落在較弱的無名指」不再是缺點:玩家跟的是畫面上的道與字母,而非哪指較強;弱指練習本就是打字訓練的目的。垂直方向的自然(家排 row1)也已對到鍵盤家排。

## 被否決的替代方案

- **人體工學/定位鍵優先**(自然欄→食指 F/J):在「一手 4 欄↔4 指」下無法保留空間順序,會使畫面上的道與鍵位錯開,破壞「鍵盤形高速公路」的價值;且各首歌欄位分佈不同,此優化本就脆弱。

## 連帶約束

- 3D 渲染(issue 06)必須採「每指一道、照鍵盤排」的版面,否則本映射的價值消失。此決策綁定渲染模型。
