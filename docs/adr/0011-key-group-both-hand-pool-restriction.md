# 鍵群:僅雙手對稱的鍵池子集(不做單手、不做自由組合)

**Status:** accepted

**鍵群(Key Group)** 是 `compileChart` 的 config 輸入(`CompileConfig.keyGroup`),把鍵指派的可用鍵池限縮到一個**預設子集**,供針對性打字練習。實作 = `buildPool(hand, keyGroup)` 依「排/指過濾」回傳較小的池,`assignKeys` 其餘機制(教學權重 D'Hondt 攤平、可玩性硬底線、determinism)原封不動。刻意約束:鍵群**一律雙手對稱**(兩手都保有鍵)、且僅開放一小組**固定預設**(`all` / `home` / `home-top` / `index-middle` / `ring-pinky`),不支援單手、不支援任意勾選。

## 為何記錄

這條範圍界線違反直覺、且會擋掉未來「順手加功能」的衝動:

1. **不做單手鍵群**:初看「只練左手」是很自然的需求,issue 原本也列了。但實測需求為「正常練習用鍵群都有雙手」;而單手會逼出「顏色→手」的衝突(某手在群內無鍵),得在「回退整池 / 鍵群凌駕顏色 / 濾掉另一手音符」間裁決,每條都動到 CONTEXT「手」語義或 judge 準確率分母。**堅持雙手對稱 → 這整個衝突不存在**:永不空手池、不濾音符、不改分母、顏色→手不變。
2. **不做自由組合**:不開放任意 `{banks, fingers}` 或逐鍵勾選,只給固定列舉。避免組合爆炸與極小池的可玩性崩壞,也讓 fixtures 可窮舉。

## 關鍵推理

- 以「縮小 `buildPool` 產出」實作,現有的可玩性降級(同手手指全被佔 → `candidates = pool`)天然變成「在鍵群內放寬」——不會借群外鍵,取捨自動正確。
- determinism 延伸涵蓋鍵群維度:同一 (譜+難度+鍵群) 永得同一組指派;無 RNG。
- `all` 為預設且等同無限制 → 與 issue 11 現況位元級相同,不回歸既有分布/覆蓋成果。
- 鍵群影響**編譯**,故 UI 落在難度選擇畫面(編譯前),非 highway 內的即時滑桿;highway 零改動(只渲染 chart 出現的鍵)。

## 被否決 / 排除

- **單手鍵群(只左手/只右手)**:見上;連帶排除為它而生的「鍵群凌駕顏色」與「濾掉另一手音符」兩種機制。
- **任意 `{banks, fingers}` / 逐鍵自訂鍵群**:無此需求,徒增複雜與可玩性風險。
- **逐字母字表 / 單字練習**:那是詞庫不是鍵群,out of scope。
- **依表現自適應調整鍵群**:本 issue 只做玩家手動選。

## 連帶影響

- `CompileConfig` 加 `keyGroup?`;`assignKeys`/`buildPool` 收之。keyAssignment 仍純函式、fixtures 可測、無新 I/O。
- 持久化:設定層(issue 12)由「純數值」擴成「數值 + 列舉」,鍵群存同一 `beat-typer:settings` blob,值不合法回退 `all`。
- CONTEXT 新增「鍵群」詞;「手」定義不變。
