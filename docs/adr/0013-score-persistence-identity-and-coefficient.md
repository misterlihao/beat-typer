# 成績持久化:難度檔雜湊為身分 + 鍵群係數折算成單一最佳

**Status:** accepted

跨場記住每張譜的最佳成績(issue 18)。三個綁在一起的決策:

1. **歌曲身分 = 選定難度檔原始文字的雜湊**(djb2/FNV → base36,純函式)。不是 Info.dat、不是 BSR code、不是 songName 組合。
2. **每張譜只存一筆最佳,不分鍵群**;跨鍵群靠**調整後準確率 = 原始 × 鍵群係數**比較。係數用仁慈壓縮曲線 `0.5 + 0.5 × (每手鍵數 / 15)`(全鍵 1.0、家+上 0.83、食中 0.80、無名小 0.70、家排 0.67)。
3. **最大 combo 與全連只在全鍵(`all`)下解鎖計入**;準確率則各鍵群折算後共爭一個最佳。紀錄 = `{ bestRawAccuracy, bestKeyGroup, bestMaxCombo, everFullCombo }`,調整值即時導出。

## 為何記錄

一旦玩家在某套鍵下累積成績,改身分方案或計分規則就會**孤兒化**既有紀錄(高逆轉成本);且這幾個選擇都違反直覺、各有真實取捨,未來讀者會問「為什麼」。

- **身分用難度檔而非 Info.dat**:成績是「打這張譜」掙的,難度檔(音符)才是那張譜。難度檔雜湊天生 per-難度(不必把難度組進鍵)、跨來源位元一致、內容真實(remap 重計、改名保留)。Info.dat 含大量與身分無關且易被編輯器重存改動的欄位。
- **不分鍵群、改用係數折算**:per-鍵群分開會讓賽前/結算要並列多組最佳、日後維護麻煩(issue 19 尤甚)。單一最佳 + 係數把「小鍵群較易」折進分數,一個數字定高下。
- **係數用壓縮曲線而非鍵數線性比例**:線性(家排 5/15=0.33)在低端太重,家排打滿只記 33%、顯示 D 級,會勸退針對性練習。`0.5+0.5×frac` 保留「全鍵最高、小群打折」的排序又不羞辱。
- **combo/FC 只全鍵解鎖**:小鍵群鍵少,連段與全連太好拿;只認全鍵的 combo/FC 才有意義。準確率因已被係數折算,故仍可各群共爭。

## 關鍵推理

- 純函式接縫獨立成 `src/scores/`(`songKey` / `adjustedAccuracy` / `updateRecord` / `coerceScores`),不塞進 compile/judge——身分與計分持久化是另一關注點。I/O(load/save 單一 localStorage blob `beat-typer:scores`)為薄層,沿用 issue 12 的 coerce 容錯 + version 遷移路數。
- 寫入時機:歌自然播畢(`player.onEnded`)。highway 對身分/儲存無感,經 `onComplete(summary)` 回呼交給 main(閉包綁 `songKey(diffText)` + 鍵群)寫入並回傳最佳供結算面板顯示。
- `updateRecord` 單調:`adjusted > 存 → 換原始+鍵群`;`all` 下 `maxCombo=max`、`everFC||=`。prev 可 undefined(首玩)。
- 係數由「每手鍵數」導出而非寫死表 → 鍵群增減自動重算,與 ADR 0011 的鍵池單一真相一致。

## 被否決 / 排除

- **Info.dat 雜湊 / BSR code 優先 / songName+audio 組合**當身分:見上;分別敗在「非內容真實且易被重存改動」「需拉線且跨來源不一致」「同名撞鍵」。
- **per-鍵群分開存**:維護與顯示麻煩,改用係數折算單一最佳。
- **鍵數線性比例係數**:低端太懲罰。
- **combo/FC 跨鍵群計入**:小鍵群太好拿,失去意義。
- **存準確率最佳的整場快照**:會弄丟獨立的全連成就(見 issue 09 grill 的同型陷阱);改存各自單調最佳。
- **成績淘汰/上限**:紀錄極小,localStorage 容量無虞,不做。

## 連帶影響

- issue 19(賽前畫面)消費最佳成績當「目標」;本 ADR 存 `bestKeyGroup + bestRawAccuracy` 即為讓 19 能顯示有脈絡的「100% 家排(計分 67%)」。
